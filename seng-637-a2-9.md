## 1. Introduction

This report presents a systematic, requirements-based unit testing strategy applied to selected components of the JFreeChart framework. The purpose of this assignment is to design and implement black-box test cases derived strictly from the Javadoc API specifications, without consulting implementation details. By restricting test derivation to documented requirements, the testing process remains specification-driven and avoids implementation bias, ensuring that the developed tests validate contractual behavior rather than internal logic.

The system under test (SUT) is the `org.jfree.data` package of JFreeChart, an open-source Java framework that provides data structures and utilities for chart construction and manipulation. For this assignment, all five public methods of `org.jfree.data.DataUtilities` were tested, along with five representative methods selected from the fifteen methods available in `org.jfree.data.Range`. These classes were chosen because they expose clear functional requirements and diverse input domains, making them well-suited for structured black-box analysis.

The testing strategy follows formal black-box test design methodologies discussed in the course, particularly Equivalence Class Testing (ECT) and Boundary Value Analysis (BVA). For each method, independent input variables were first identified, and their respective domains were defined according to the Javadoc specifications. These domains were then partitioned into valid and invalid equivalence classes, followed by the identification of boundary values for numerical and structural parameters. Test cases were designed on paper prior to implementation to ensure traceability between the documented test strategy and the resulting JUnit test suite.

Several methods in `DataUtilities` accept interface types such as `Values2D` and `KeyedValues`. To preserve strict unit isolation and avoid dependency on unknown implementations, mocking was employed to simulate controlled behavior for these interfaces. This allowed precise control over return values and boundary conditions while maintaining a pure unit-testing perspective.

It is important to note that the provided versions of JFreeChart include intentionally seeded defects. Therefore, test oracles were derived exclusively from the specifications rather than observed runtime behavior. This reinforces the principle that unit testing should evaluate conformance to documented requirements, even when the implementation may be faulty.

Overall, this report demonstrates a disciplined and coverage-oriented approach to black-box unit testing, emphasizing systematic partitioning, boundary reasoning, traceability, and professional testing practices consistent with industry standards.

## 2. Detailed Description of Unit Test Strategy

The unit test strategy was developed using structured black-box testing techniques. All test cases were derived from the Javadoc specifications, and no implementation details were consulted during the design process.

For each method under test, the following approach was applied. First, independent input variables were identified from the method signature. The valid and invalid input domains for each variable were then defined based on the documented behavior. These domains were partitioned into equivalence classes representing distinct categories of inputs. Where numerical or structural limits were present, boundary values were identified to ensure that edge conditions were properly tested.

After defining partitions and boundaries, test cases were constructed to ensure that each equivalence class and boundary condition was covered at least once. The goal was to achieve complete coverage in a clear and organized manner, without creating unnecessary or redundant tests.

Each method is presented below with a brief explanation of the input analysis, followed by tables summarizing equivalence classes and boundary values.

## 5. Difficulties Encountered and Lessons Learned

During the development of the test suite, several practical and conceptual challenges were encountered. One of the primary difficulties was strictly avoiding implementation bias. Since the JFreeChart versions provided for the lab contained seeded defects, it was important to resist the temptation to adjust test cases based on observed runtime behavior. All expected outcomes had to be derived solely from the Javadoc specifications, even when the implementation behaved differently. Maintaining this discipline required careful attention to documented requirements and consistent cross-checking against the API descriptions.

Another challenge involved correctly identifying all independent input variables for each method. Some methods, particularly in `DataUtilities`, contained multiple interacting parameters (e.g., object references and index values). Ensuring that each variable's domain was defined clearly and partitioned independently required deliberate analysis before constructing test cases. Missing or incorrectly identifying an independent variable would have resulted in incomplete partition coverage.

Floating-point boundary analysis also required careful consideration. Because many methods operate on double values, it was necessary to account for precision tolerances when comparing expected and actual results. Determining appropriate boundary values and ensuring correct assertion tolerances in JUnit demanded attention to detail.

Additionally, avoiding redundant test generation was a non-trivial task. While striving for complete coverage of equivalence classes and boundary values, it was important not to create overlapping or unnecessary test cases. Balancing thoroughness with efficiency required careful planning before implementation.

Through addressing these challenges, several key lessons were learned. First, structured equivalence class partitioning significantly improves clarity and reduces ambiguity in test design. By defining input domains formally before writing code, the testing process becomes more organized and defensible. Second, boundary value analysis frequently reveals subtle edge behaviors that may not be apparent from normal-case testing alone. Finally, maintaining explicit traceability between the documented test strategy and implemented JUnit methods strengthens confidence in coverage completeness and improves the overall quality of the test suite.
